## Context

The system generates short motion-graphic videos from natural language prompts. An LLM produces a JSON video description (nodes with types, positions, animations) that a deterministic canvas renderer draws frame-by-frame. The JSON schema is validated with Zod.

**The core problem**: The AI is given a 960×540 pixel canvas and asked to compute raw `(x, y)` coordinates for every element. It must mentally model anchor semantics, group-relative positioning, dimension-dependent centering, and inter-element spacing — all as absolute pixel math. It fails unpredictably:

1. **Text centering is broken by design**: `getNodeDimensions` returns `width: 0` for text without `maxWidth`, so `getAnchorOffset` with `anchor: "center"` computes a zero-width offset — the text draws from its x/y point rather than centering on it.
2. **No layout abstractions**: To vertically stack 8 rectangles with spacing, the AI must compute 8 `(x, y)` pairs manually. It gets these wrong regularly, producing overlapping or clipped elements.
3. **The prompt is prose, not a contract**: A ~90-line English essay describes rules the AI should follow. This diverges from the actual schema, can't be validated, and gives the AI too much freedom to hallucinate interpretations.
4. **No visual validation**: A node at `x: 2000` on a 960-wide canvas passes Zod validation — the system has no concept of "this will be invisible."
5. **Prompt/schema mismatch**: The prompt says "anchor defaults to top-left when omitted" but the runtime default in `getBaseNodeValues` is `"center"`. The AI computes positions for one convention; the renderer applies another.

**Learnings from json-render.dev**: Their architecture solves the same class of problem (AI generates JSON → components render it) with three key patterns we lack:
- **Catalog**: A formal registry of component types with Zod-typed props and descriptions, from which the system prompt is auto-generated.
- **Design-token props**: `z.enum(['sm', 'md', 'lg'])` instead of raw pixel values — the AI picks from constrained choices; the component decides what that means.
- **Structural validation**: `validateSpec()` catches invalid references and unknown types before rendering.

## Goals / Non-Goals

**Goals:**
- Make AI-generated video layouts correct by default for common patterns (centered elements, vertical/horizontal stacks, titled graphs).
- Eliminate raw pixel math from the AI's responsibilities wherever a layout primitive can replace it.
- Auto-generate the system prompt from code so it cannot diverge from what the renderer actually supports.
- Catch visually broken output (off-screen, zero-dimension, misanchored) before rendering.
- Fix the text node dimension model so anchor-based centering works.

**Non-Goals:**
- Full CSS-like layout engine (flexbox, grid) — we only need a few composable primitives for video motion graphics.
- Changing the rendering pipeline (skia-canvas, frame-by-frame) — this is about what JSON the AI generates, not how pixels are drawn.
- Streaming/patching (json-render's JSONL streaming) — our videos are generated in one shot, not streamed progressively.
- Changing the AI model or provider — the fix is architectural, not model-dependent.

## Decisions

### 1. Catalog system that auto-generates the prompt

**Decision**: Create a `defineCatalog()` function that declares each node type with its Zod prop schema, slot declarations, and natural language descriptions. The system prompt is generated by `catalog.toPrompt()` rather than hand-written.

**Why not keep the hand-written prompt?** The current prompt already has a factual error (anchor default documentation) and will inevitably diverge further as node types are added. Auto-generation from the source-of-truth schema eliminates this class of bug entirely.

**Why not use json-render directly?** json-render is built for UI rendering (React components, DOM). Our renderer is a canvas-based video pipeline. The *pattern* transfers; the library doesn't.

**Shape:**
```typescript
const catalog = defineCatalog({
  nodes: {
    rect: {
      props: videoRectPropsSchema,    // Zod schema (the source of truth)
      slots: [],                       // leaf node — no children
      description: "Rectangle shape with optional fill, stroke, and corner radius.",
    },
    stack: {
      props: videoStackPropsSchema,
      slots: ["children"],
      description: "Vertical or horizontal arrangement of child nodes with automatic spacing.",
    },
    // ...
  },
  primitives: videoPrimitiveSchema,    // existing animation primitives
  anchors: videoAnchorSchema,
  easings: videoEasingSchema,
});
```

### 2. Layout primitives: `stack`, `center`, `align`

**Decision**: Add three layout node types that the AI uses instead of raw pixel coordinates:

| Primitive | What it does | AI specifies | Renderer computes |
|-----------|-------------|--------------|-------------------|
| `stack` | Arranges children vertically or horizontally | `direction`, `gap`, `align` | Child `(x, y)` positions |
| `center` | Centers a single child in the frame (or in a parent) | Just the child | `x = parentWidth/2`, `y = parentHeight/2` |
| `align` | Positions a child relative to a named edge/corner | `position` (e.g. `"top-center"`, `"bottom-right"`), `padding` | Absolute `(x, y)` from frame dimensions |

**Why these three?** They cover the patterns that fail most often: centering (the square example), vertical lists (the Riemann bars), and title placement. They compose: a `stack` inside a `center` inside an `align` handles "centered column of items at the top of the frame."

**Why not full flexbox?** Diminishing returns. These three primitives cover ~80% of the compositions the AI attempts. Adding `flex-wrap`, `flex-grow`, `justify-content` etc. would make the schema harder for the AI to use correctly and harder for us to implement in a canvas renderer.

**Alternative considered — virtual coordinates (0-1 range):** Instead of pixels, the AI would use fractional coordinates. Rejected because it doesn't help with inter-element spacing (the main failure mode) and adds a mental model translation layer that makes absolute-position cases harder.

### 3. Fix text node dimensions

**Decision**: Text nodes gain a computed `width` in the renderer based on actual text measurement. When `maxWidth` is set, that's the width for anchor math. When it's not set, the renderer measures the text at the given font size and uses that measurement.

**Why not require `maxWidth`?** It forces the AI to estimate pixel widths for text, which is exactly the kind of pixel math we want to eliminate. The renderer has access to the canvas context and can measure precisely.

**Implementation**: In `getNodeDimensions` for text nodes, use `context.measureText()` when `maxWidth` is not provided. This requires threading the canvas context through to anchor calculation, or pre-computing text widths in a resolution pass.

### 4. Visual validation pass

**Decision**: After Zod schema validation succeeds, run a visual validation pass that checks:
- **Bounds**: Nodes with known dimensions are not entirely off-screen (x + width < 0, x > canvasWidth, etc.)
- **Dimension sanity**: No node has zero effective area (width or height ≤ 0 after animation resolution at frame 0)
- **Layout resolution**: Layout nodes (`stack`, `center`, `align`) resolve to valid absolute positions before rendering

These produce warnings (not hard errors) because edge cases exist (intentionally off-screen elements animating in). The system logs them and optionally surfaces them to the user.

**Alternative considered — hard rejection:** Rejected because some valid animations start off-screen. Warnings let us catch bugs without blocking creative use.

### 5. Prompt structure: component docs, not prose rules

**Decision**: The auto-generated prompt follows this structure:

```
1. Output format: JSON matching the provided schema
2. Canvas: {width}×{height} @ {fps}fps
3. Available components:
   - rect: {description}. Props: {from Zod schema}
   - text: {description}. Props: {from Zod schema}
   - stack: {description}. Props: {from Zod schema}
   ...
4. Layout guidance:
   - Use `center` to place an element at the frame center
   - Use `stack` for lists or sequences of elements
   - Use raw x/y only when precise positioning is needed
5. Animation: {primitives list}, {animate block format}
6. Constraints: {timing, id uniqueness, etc.}
```

Each component section is generated from the catalog. New node types automatically appear in the prompt. The layout guidance section is the only hand-written part.

## Risks / Trade-offs

**Risk: Layout primitives increase schema complexity** → Mitigation: Layout nodes are optional. The AI can still use raw `x/y` for precise positioning. The prompt guides it toward layout primitives for common patterns but doesn't force them.

**Risk: Text measurement requires canvas context during validation** → Mitigation: Use a lightweight text measurement approach (font metrics table or a single shared canvas context) rather than creating a full canvas per validation call.

**Risk: Existing generated videos break** → Mitigation: Layout nodes are additive — all existing node types and their positioning continue to work identically. The text dimension fix is the only behavioral change, and it only affects text nodes with `anchor: "center"` and no `maxWidth` (which were already rendering incorrectly).

**Risk: Catalog maintenance overhead** → Mitigation: The catalog is derived from the existing Zod schemas, not a separate definition. Adding `description` strings and `slots` declarations to existing schemas is minimal overhead compared to maintaining a separate prompt document.

**Risk: AI may still produce bad layouts** → Mitigation: The visual validation pass catches the worst cases. Layout primitives reduce but don't eliminate the possibility of bad output. This is defense-in-depth, not a silver bullet.
